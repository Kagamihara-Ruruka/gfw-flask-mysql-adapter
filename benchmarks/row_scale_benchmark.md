# Common Adapter row-scale benchmark

測試日期：2026-06-11

## 測試目的

這份 benchmark 只測「展示 pipeline」：

1. 前端送出查詢。
2. Flask adapter 透過 PyMySQL 查 MySQL。
3. API 回傳 JSON。
4. 前端把資料渲染到 Leaflet 地圖與 HTML table。

不測資料清洗。資料清洗視為上游前處理責任。

## 測試資料

母表：legacy local DB 的 `gold_grid` table

總 row 數：2,840,514

## 測試條件

- 查詢日期：`2024-01-01`
- 前端 limit：5,000 rows
- 表格預覽：300 rows
- 地圖渲染：5,000 Leaflet rectangle paths
- 每個比例跑 3 次，表格採 median
- 自動化 benchmark 阻擋外部底圖 tile 請求，避免把第三方地圖服務與網路波動算進本機資料展示 pipeline
- 耗時單位一律換算為秒

## 判讀

在目前查詢條件下，資料表從 1% 放大到 100% 後，前端完整路徑 median 仍落在約 0.1400 s 到 0.1940 s。這表示目前 MVP 的主要限制不是 MySQL 表總 row 數，而是單次回傳與前端渲染的 row 數。

目前展示上限 `limit=5000` 是合理防線。若未來要提高上限，應先測 Leaflet path 數量與 DOM table 預覽策略，而不是先改 SQL。
