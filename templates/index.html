播放交付策略是目前唯一控制時間軸語意的入口。分析模式會保證每張快照都被用上；流暢與嚴格模式端口已保留，但在實作完成前不接管時間線。查詢、渲染與模糊不應反過來控制這條時間軸。

資料快取 / 預熱

這一組只管資料供應：完整預熱會先走 range packet；短緩衝會用逐日期背景預載補前方窗口；逐張查詢不主動預載，但仍會讀既有快取。日期上限為 0 時不限制；並行預熱數只控制背景預載；快取上限用來避免一次預熱過大範圍造成記憶體壓力。

緩衝中

Frame 補間

目前支援播放時的視覺補間 policy：Layer crossfade 只修飾兩張真實 snapshot 的 layer 替換；關閉時播放換幀會直接切換。真正資料補間會是獨立的 requestAnimationFrame 循環，只在兩張已 ready 的 rows/render artifact 之間計算 alpha，不查 SQL、不推進真實日期，也不取代 BI 探針需要的 rows packet。

視覺效果

淡入淡出只修飾圖層替換；高斯模糊只作為縮放 / LOD 重算時的短暫遮罩，不參與播放時間線，也不等待補間。

渲染壓力

這條時間軸未來用來收斂 WebGL/Canvas、render artifact cache、補間 FPS 與跳幀壓力。現階段仍由硬體加速狀態機與 GFW renderer registry 自動選擇繪圖路徑。

測速觀測

儀表板的「測速」box 是目前的觀測面：SQL、API、client fetch-to-render、renderer draw、row count 與 render-state 都在那裡更新。這裡只宣告責任邊界，避免把測速數值混成播放控制。
